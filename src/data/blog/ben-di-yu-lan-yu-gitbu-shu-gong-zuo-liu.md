## 📍 阶段一：本地环境的觉醒（启动发开服务器）

在发布到广阔的互联网之前，我们需要在自己的“实验室”里观察代码的变化。这就需要启动 Astro 的本地开发服务器。

### 核心步骤与命令
1. **进入实验区 (Change Directory)**：通过终端指令精准定位到数字花园的根目录结构。
   ```bash
   cd "/Users/momo/momoworks/2025 Obsiddian/Moxiaoxia@TCM Museum/TCM_Library/TCM_Blog"
   ```
2. **启动引擎 (Run Development Server)**：告诉由 npm 驱动的后台引擎开始工作。
   ```bash
   npm run dev
   ```
3. **实时观测 (Live Preview)**：引擎启动后，终端将释放一个本地信号（如 `http://localhost:4321`）。在浏览器中打开此信号，即可在安全隔离的本地环境中观察代码修改带来的实时映射变化。

## 📍 阶段二：定格进化与云端同步（Git 提交流水线）

当我们在本地完成了满意的内容创作或代码调试后，需要将这些“进化”打包，通过版本控制系统（Git）安全地传送至云端仓库（GitHub）。

### 核心步骤与命令
1. **打包所有的改动 (Stage All Changes)**：将当前所有有修改的文件放入暂存区，准备打上时代烙印。
   ```bash
   git add .
   ```
   *(注：`add` 与 `.` 之间的空格不可省略，`.` 代表无差别地包含当前目录下的所有演化)*

2. **打上进化标签 (Commit Changes)**：附带严谨且富于意义的说明，清晰记录这次代码演化的方向。此条遵循 `03_OPERATING_PROCEDURES.md` 关于每次提交需清晰标注的原则。
   ```bash
   git commit -m "类型: 更新了首屏内容，注入了洞穴博物馆的核心愿景与人物称谓修正"
   ```

3. **推送至远端星系 (Push to Remote)**：将包裹好的变动发往远端的 GitHub 星球（主分支）。
   ```bash
   git push origin main
   ```
   *(注：将本地 `main` 分支的内容推送至名为 `origin` 的远程服务器上)*

## 📍 阶段三：从沉眠到重生的 Vercel 自动部署

**魔法的发生：**
当我们执行完 `git push` 命令、代码抵达 GitHub 的那一刻，Vercel 就像一个不知疲倦的数字灯塔，迅速在黑暗中捕捉到了这一更新信号。

- **无需额外指令**：我们不需要登录 Vercel 或运行额外的上传指令。
- **自动化构建**：Vercel 的 CI/CD（持续集成与持续部署）系统会自动拉取最新版的代码，执行构建动作，将其转化为全世界可访问的纯静态页面。
- **全网生效**：一至两分钟后，博客绑定的公网域名将自动切换至这个最新生成的版本。
